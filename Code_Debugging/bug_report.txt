**Bug Report and Fixes**

### 1. Summary of Issues Found
Three bugs were identified in the `factorial(n)` function:
* Missing base case to stop recursion
* No error handling for negative input
* No input validation for non-integer input

### 2. Detailed Explanation of Each Fix
* **Bug 1: Missing Base Case**: A base case was added to handle the case when `n` is 0 or 1, returning 1 in these cases to stop the recursion.
* **Bug 2: No Error Handling for Negative Input**: Input validation was added to check if `n` is a non-negative integer, raising a `ValueError` if `n` is negative.
* **Bug 3: No Input Validation for Non-Integer Input**: Input validation was added to check if `n` is an integer, raising a `TypeError` if `n` is not an integer.

### 3. Impact Assessment
The fixes ensure that the `factorial(n)` function:
* Does not exceed the maximum recursion depth, preventing a `RecursionError`
* Handles negative input correctly, preventing infinite recursion
* Produces correct results for non-negative integer input
* Raises informative errors for invalid input

### 4. Alternative Solutions Considered
Alternative solutions considered include:
* Using an iterative approach instead of recursion to calculate the factorial
* Using a library function to calculate the factorial, such as `math.factorial()` in Python
* Implementing a more robust input validation system to handle a wider range of input types and values.

**Corrected Code**
```python
def factorial(n):
    if not isinstance(n, int):
        raise TypeError("Input must be an integer.")
    if n < 0:
        raise ValueError("Input must be a non-negative integer.")
    if n == 0 or n == 1:
        return 1
    return n * factorial(n - 1)

print(factorial(5))
```
